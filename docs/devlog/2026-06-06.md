## i18n: Übersetzungslücken im DNA- und Vote-Tab

Im DNA-Tab fehlten bei bestimmten Lokalen Übersetzungen für den Bestätigungs-Flow beim Senden von Votes: der Senden-Fortschritt ("Sende Votes… x/x"), der Bestätigungstext, die Nachhaltigkeits-Labels und die Scan-Zusammenfassung. 32 neue Keys wurden in alle 14 Sprachen integriert.

Dabei wurden drei systmatische Fehler in den Lokalen HI, BN und JA korrigiert: `recoveryBodyNoReduction` und `whaleExplainBody` enthielten je eine falsche Sprache, weil die Einträge in der Datei eine Position nach vorne verschoben waren. Der Fehler war in der Dateistruktur begründet und betraf nur diese beiden Keys.

Die Landingpage beschrieb "Community Pools" als Feature, das nicht existiert. Der Eintrag wurde durch "Author Discovery" ersetzt — eine Funktion die tatsächlich vorhanden ist und auf Whale-Signalen und Trust Scores basiert.

## Regression: Vote-History schlägt für neue Accounts fehl

`testuser1` hat rund 570 Gesamtoperationen in der Chain-History. Beim Abrufen der Vote-History iteriert der Backend-Loop in Batches von je 100 Einträgen rückwärts. Im letzten Batch fiel `fromSeq` auf 66 — aber `limit` blieb auf 100. Die Steem API verlangt `start > limit` (strikt), wirft bei 66 < 100 einen Fehler, und der gesamte DNA-Tab zeigt einen roten Fehlerblock.

Betroffen sind alle Accounts mit weniger als circa 600 Gesamtoperationen — also genau die neuen Nutzer, die als erste testen.

Fix in `voteHistory.ts`:

```typescript
const limit = fromSeq < 0
  ? MAX_OPS_PER_BATCH
  : Math.min(Math.max(fromSeq - 1, 0), MAX_OPS_PER_BATCH);
if (fromSeq >= 0 && limit <= 0) break;
```

Während der Fehlersuche stellte sich heraus, dass alle bisherigen API-Deployments an den falschen Container-Pfad kopiert wurden: `/app/dist/` statt `/app/apps/api/dist/`. Der Node.js-Prozess läuft aus letzterem. Die Architektur-Dokumentation und der Deploy-Workflow wurden korrigiert.

## Onboarding: Sackgassen beseitigt

Wenn die Vote-History fehlschlug oder leer war, hat der DNA-Tab einen Fehlerblock angezeigt und dabei den Strategie-Editor vollständig ausgeblendet — obwohl beides konzeptionell unabhängig ist. Wer als neuer Nutzer noch keine Votes hat, konnte weder die DNA-Analyse nutzen noch manuell Autoren hinzufügen.

Die Lösung: `InlineStrategyEditor` als eigenständige Komponente, die in beiden Zuständen sichtbar bleibt. Im Fehlerfall erscheint sie unterhalb einer kompakten Fehlermeldung mit Retry-Button. Im Empty-State erscheint sie unterhalb einer Willkommens-Box, die direkt zum Community-Tab verlinkt.

Der neue Pfad für Accounts ohne Vote-History:

```
Login → DNA-Tab: Willkommen + Community-Tab öffnen → Autoren entdecken
→ Zur Strategie hinzufügen → Plan generieren → Votes senden
```

Kein Dead End mehr an keiner Stelle dieses Pfades.

## Auth: Active Key nie wieder

`buildAuthorityGrantUrl()` in `steemConnectConfig.ts` setzte `authority=active`. Das bedeutete: wer @votebroker die Posting Authority erteilen wollte, wurde zu SteemLogin weitergeleitet und dort nach dem Active Key gefragt. VoteBroker braucht ausschließlich Posting Authority, und das war von Anfang an die Intention.

Geänderter Parameter: `authority=active` → `authority=posting`.

Gleichzeitig wurden in allen 14 Lokalen die `authorityNote`-Texte korrigiert. Bisher stand dort explizit "Active Key bereithalten" — was die Verwirrung aktiv verstärkt hat. Neu: "Posting Key genügt, kein Active Key".

## Steem Keychain Integration — drei Phasen

[@michelangelo3](https://steemit.com/@michelangelo3) empfahl in seinem Bericht SteemKeychain als Alternative zu SteemLogin. Aus seinem Thread meldete sich anschließend [@faisalamin](https://steemit.com/@faisalamin), Entwickler von Steempro und Maintainer der aktuellen Steem Keychain Extension (v3.0.4), mit einem konkreten Hinweis zur API-Version.

Nach einem Blick in den Quellcode der Extension waren alle nötigen Informationen vorhanden, um alle drei Integrations-Phasen heute umzusetzen.

**Phase 1 — Authority Grant via Keychain**

Wenn Keychain im Browser installiert ist, ersetzt `requestAddAccountAuthority` den SteemLogin-Redirect für die Posting Authority. Der Nutzer sieht einen Keychain-Dialog, bestätigt mit dem Posting Key, und die Authority ist gesetzt. Kein Tab-Wechsel, kein fremder Domain-Trust.

```
Keychain vorhanden → requestAddAccountAuthority(@votebroker, "Posting", 1) → fertig
Kein Keychain      → SteemLogin /import?authority=posting (Fallback, bereits auf Posting-Key korrigiert)
```

**Phase 2 — Vote-Signing via Keychain**

Votes werden vom Nutzer selbst signiert und direkt an den Steem-Node gesendet, statt über den Server geroutet zu werden. Das Backend erhält danach nur noch die TransactionID zur Audit-Log-Erfassung.

```
Keychain vorhanden → requestVote() → txId → POST /api/votes/execute?mode=keychain (Audit)
Kein Keychain      → POST /api/votes/execute (Server-Vote, Fallback)
```

Für Keychain-Nutzer entfällt damit die Notwendigkeit, @votebroker überhaupt eine Posting Authority zu erteilen.

**Phase 3 — Login via Keychain**

Der initiale Login läuft jetzt ebenfalls ohne SteemLogin-Redirect, wenn Keychain vorhanden ist. Ablauf:

1. Nutzer gibt seinen Steem-Benutzernamen ein
2. Backend erzeugt eine zufällige 64-Byte Nonce (TTL 5 Minuten)
3. Keychain signiert die Nonce mit dem Posting Key (`requestSignBuffer`)
4. Backend verifiziert die Signatur gegen den Public Posting Key des Accounts aus der Chain
5. Bei Erfolg: Session wird erstellt — identisch zur bisherigen SteemConnect-Session

```typescript
const recoveredKey = Signature.fromString(signature).recover(sha256(nonce)).toString();
const valid = account.posting.key_auths.map(([k]) => String(k)).includes(recoveredKey);
```

SteemLogin bleibt als vollständig funktionierender Fallback erhalten — visuell zurückgestuft, aber gleichwertig.

---

## Stand nach heute

Nutzer mit Keychain interagieren an keiner Stelle mehr mit steemlogin.com: nicht beim Login, nicht beim Authority-Grant, nicht beim Voten. SteemLogin ist reiner Fallback für alle anderen.

Der erste Tester hat das Produkt schlechter getroffen als es war — durch eine Regression im Vote-History-Loop und durch fehlende Führung im Onboarding. Beides ist beseitigt. Aus dem Feedback ist außerdem eine vollständige Keychain-Integration entstanden, die das Login-Vertrauen für erfahrene Steem-Nutzer deutlich verbessert.
