# DevLog 2026-06-19 — Shadow Model v4: eigenständiges, de-confoundetes Curation-Modell

## Ziel

Kein Re-Tuning von v3. Ein **eigenständiges** experimentelles Curation-Modell, das
die tatsächliche Qualität einer Opportunity schätzt — parallel zu v3 im Shadow-Mode,
ohne je einen echten Vote auszulösen, mit direktem Side-by-Side-Logging pro Kandidat.

## v4-Modell: `src/chain/opportunityScoreV4.ts`

Probabilistisch statt Punkte-Gates: Ausgabe ist **P(gute Opportunity)** ∈ [0,1]
(`pGood`), als `score` = round(pGood·100) für direktes Nebeneinander zu v3.
Entscheidung = `pGood ≥ kategorie-spezifischer Schwellwert` (weich). Harte Skips nur
bei mechanischer Wahrheit (Payout-Fenster abgelaufen) oder Nutzer-Intent (`ignorieren`).

### Vier bewusst entschärfte Confounds (Korrelation ≠ Kausalität)

1. **Growth Factor durch eigenes Timing verzerrt** → v4 nutzt GF **gar nicht**. Der
   Nenner (pending zum Vote-Zeitpunkt) hängt davon ab, wie früh *wir* voten, nicht von
   Postqualität. Ein timing-konfundiertes Feature kann Qualität nicht messen → raus,
   nicht „korrigiert“.
2. **Autorqualität ≠ Whale-Effekt** → Autorqualität aus **Median**-Payout (robust gegen
   Whale-Ausreißer). Whales nur als separater, **sättigender** Confirmation-Term (Cap 5,
   Effekt invertiert darüber laut v3-Historie).
3. **Poolgröße ≠ Postqualität** → pending payout bekommt **Gewicht 0**. v3 vergab bis 25
   Punkte allein für einen *kleinen* Pool — Quelle von 73 % seiner False Positives.
4. **VP-Einsatz ≠ Erfolg** → v4 bewertet nur Opportunity-*Qualität*, gewicht-/VP-agnostisch.
   Budget/Weight bleiben downstream beim Planner.

### Features (Logit)

`logit = bias + wAuthorQuality·(authorQ−0.5) + wWhaleConfirm·whaleConfirm
       + wConsistency·(consist−0.5) + wCommunityYield·(commY−0.5)
       + wTiming·(timing−0.5) + wReverseAuction·[age<5] + wFreshnessDecay·decay
       + wPoolNeutral·(pool−0.5)`   (wPoolNeutral = 0)

- **authorQuality** (w=2.8, stärkstes): log1p(median payout)/log1p(5)
- **whaleConfirm** (w=1.1): √(min(whaleCount,5)/5 · followRate) — braucht Whales *und* Propensität
- **consistency** (w=0.6): 1/(1+sp_per_vp_cv) — stetige Autoren minimal sicherer
- **communityYield** (w=0.4): schwacher Community-Prior
- **timing** (w=0.5, klein): Glocke Peak 15–20 min — EDA zeigt ~0 kausalen Effekt
- **reverseAuction** (w=−1.6): <5 min → Curator-Reward fließt zurück in Pool (Mechanismus, kausal)
- **freshnessDecay** (w=−2.0): Posts nahe Fensterende können keine Curation realisieren

Unbekannter Autor → neutraler Prior 0.5 (keine Bestrafung). Self-Post → von
reverse-auction ausgenommen.

### Smoke-Test auf 522 echten Signal-Autoren (frischer On-Peak-Post, normal)

`score: min 26 · p25 54 · median 70 · p75 86 · p90 91 · max 91` — differenziert, nicht
degeneriert. `would_vote @ thr 0.50: 407/522 (78 %)` — plausibel auf diesem
vor-selektierten Set (Autoren mit Whale-Signal); Schwellwerte sind Priors für die Kalibrierung.

## Self-Calibration-Hook: `src/jobs/v4Calibration.ts`

`deriveV4Calibration(db)` fittet Logit-Gewichte aus `vb_global_vote_outcomes`, sobald
genug realisierte Outcomes vorliegen (`MIN_CALIBRATION_SAMPLE = 500`). Methode:
Point-biserial-Korrelation jedes Features mit dem Label „Post ≥ 1 SBD gesettled“,
skaliert in den Logit-Raum. Bewusst **nicht-anwendend** — gibt einen *Vorschlag* zurück
(oder `insufficient_outcomes`), nichts überschreibt die Live-Gewichte automatisch.
Mechanismus-Terme (reverse-auction, freshness) und die Pool-Neutralität werden nicht aus
Korrelation re-gefittet — sie kodieren kausale/strukturelle Entscheidungen.

Auf der Live-DB heute: `{ ok:false, sampleN:0, reason:'insufficient_outcomes' }` → v4
läuft auf Priors. Nächtliche Verdrahtung ist ein Einzeiler, sobald Prod-Daten bestätigt sind.

## Logging: bestehende Tabelle erweitert (Entscheidung des Users)

Keine neue Tabelle. `vb_copilot_shadow_runs` um `v4_score`, `v4_decision`,
`v4_components`, `v4_version` ergänzt (CREATE + idempotente ALTER-Migration). v4 wird auf
**demselben Kandidaten** gescored, den v3 loggt → direkter Side-by-Side-Vergleich ohne
Join. Die Outcome-Spalten (`resolved_payout_sbd`, `outcome_status`) sind geteilt, also
gilt das realisierte Ergebnis automatisch für beide Modelle.

`v4_components` enthält zusätzlich `independentBest` (v4s eigene Top-Wahl + ob sie von
v3 abweicht) — für die Analyse „hätte v4 *bessere* Posts gefunden“. Limitation: der
*Outcome* von v4s abweichender Eigenwahl wird hier nicht aufgelöst (die Zeile gehört
v3s Pick). Volle Auswertung divergenter Picks braucht das separate Modell-Scoring-Schema
(v5/v6-Phase) — bewusst verschoben.

### Integration `copilotShadowJob.ts`

- `getWhaleSignal` um median_payout / whale_follow_rate / sp_per_vp_cv erweitert (v3 ignoriert sie).
- `getCommunityAvgPayout` neu.
- v4 scored alle eligible Posts unabhängig, loggt Verdict auf v3s Pick + eigene Top-Wahl.
- `skip_no_posts` / `skip_already_voted` → v4-Felder null (via `{ ...V4_NULL, ...row }` beim Insert).

## Nachtrag — Prod-DB gefunden, Point-in-time-Backtest, Autor-Prior

### DB-Verwechslung aufgelöst
Die „leeren Outcome-Tabellen" weiter oben basierten auf der **falschen DB**. Die produktive
DB ist das Docker-Volume `votebroker_data` (`/var/lib/docker/volumes/votebroker_data/_data/votebroker.db`,

(1.017 für @jan-philippvieth), 54k Shadow-Runs, 690k whale_details. Kein Backfill nötig.

### Point-in-time-Backtest v4 vs v3 (`scripts/backtestV4.mjs`)
Read-only gegen Prod-Volume. v4 über 4.959 aufgelöste v3-Shadow-Entscheidungen neu gerechnet;
Autor-/Whale-/Community-Features **ausschließlich aus Events < run_at** rekonstruiert
(`vb_whale_vote_details`, `vb_global_vote_outcomes`) — Live-`vb_signal_author` NICHT gelesen
(Zukunftsinfo). Label `resolved_payout_sbd ≥ 1 SBD`.

Ergebnis (Default-Prior 0,65, Gesamtset n=4.959, Base-Rate 0,759):

| Modell | Precision | Recall | F1 | FP | Ø Payout(pred+) |
|---|--:|--:|--:|--:|--:|
| v3 (wie geloggt) | 0,757 | 0,986 | 0,857 | 1.190 | 6,09 SBD |
| v4 (point-in-time) | 0,984 | 0,788 | 0,875 | 49 | 8,44 SBD |

v4 senkt FPs von 1.190 → 49 und trifft ertragreichere Posts; Preis ist Recall. **Einschränkungen:**
v3-Baseline ist die ältere Pre-17.06-Variante; aktuelles v3 nicht fair nachrechenbar, weil es
`pendingPayoutSbd` braucht (historisch 0 % geloggt) — v4 ignoriert den Pool per Design und ist
dadurch überhaupt sauber backtestbar. Autor-Historie aus März-Fenster (point-in-time gültig, aber
stale); Label = Post-Payout, nicht Curation-SP; vorgefiltertes Set, hohe Base-Rate.

### FN-Analyse + Single-Lever (`scripts/backtestV4_fn.mjs`)
924 FNs (bei Prior 0,5) gebucketed: 688 Timing (geringwertig, weit unter Schwelle, korrekt
geskippt), 118 knapp-unter-Schwelle (hochwertig, Median 20 SBD!), **114 fehlende Autorhistorie**
(gut zahlend, 10 SBD), 0 fehlende Whale-Signale. Hebel-Vergleich unter Precision-Floor 0,95:
- globaler Threshold-Shift: nur +17 FN (Precision-Kurve zu steil)
- **neutraler Autor-Prior 0,5 → 0,80: +144 FN** bei Precision 0,982 → klarer Sieger

### Umsetzung: tunbarer Autor-Prior
`unknownAuthorPrior` als Kalibrierungs-Konstante in `opportunityScoreV4.ts`. Unbekannte Autoren
erhalten statt strikt-neutral 0,5 nun einen tunbaren Prior. **Default 0,65** (moderat — das volle
Opportunity-Universum hat mehr schlechte Unbekannte als das vorgefilterte Backtest-Set, daher
unterhalb des 0,80-Optimums; Live-Kalibrierung soll feinjustieren). **Runtime-konfigurierbar** via
env `V4_UNKNOWN_AUTHOR_PRIOR`, per-Call über `V4Calibration`-Objekt überschreibbar. Version → `v4.1-priors`.
Effekt: Recall 0,755 → 0,788, F1 0,855 → 0,875, Precision ~unverändert (nur Unbekannte betroffen,
High-Fidelity-Subset bleibt 0,998).

### Shadow-Logging erweitert
`vb_copilot_shadow_runs` + `author_prior_used` (REAL, angewandter Prior oder NULL bei echter
Historie) und `author_history_available` (INTEGER 1/0). `V4Result` trägt beide als
`authorPriorUsed`/`authorHistoryAvailable`. Erlaubt Recall-Verlust-Analyse direkt aus der Tabelle.
