# DevLog 2026-06-22 — Refactoring Phase 1: kanonische Post-Outcome-Tabelle

Start der Entflechtung dreier fachlicher Ebenen, die heute in
`vb_global_vote_outcomes` vermischt sind: **Post-Outcome** (Eigenschaft des Posts,
einmal pro `(author, permlink)`), **Vote-Outcome** (Eigenschaft des Votes, pro
`(voter, author, permlink)`) und **Modell-Vorhersage** (pro Modell). Details und der
Phase-2/3-Plan in [ADR-006](../decisions/ADR-006-outcome-prediction-separation.md).

Phase 1 liefert nur die kanonische Post-Outcome-Tabelle — strikt additiv.

## Was umgesetzt wurde

- **Neue Tabelle `vb_post_outcomes`** (`db/index.ts`, `CREATE TABLE IF NOT EXISTS`):
  Schlüssel `(author, permlink)`, Felder `post_created_at, community, final_payout,
  net_votes, active_votes, paid_at, resolved_at`. Genau ein Outcome pro Post.
- **`apps/api/src/chain/postOutcomes.ts`:**
  - `upsertPostOutcome` — kanonischer Merge-Writer. Stabile Identitätsfelder behalten
    ihren ersten Wert; aufgelöste Messwerte (`final_payout`, `net_votes`,
    `active_votes`) nehmen den neuesten Nicht-NULL-Wert; `resolved_at` wird beim ersten
    bekannten `final_payout` gestempelt.
  - `syncPostOutcomesFromGvo` — Backfill + idempotenter Reconcile per reiner SQL-
    Aggregation aus `vb_global_vote_outcomes` (`GROUP BY author, permlink`).
    **Kein** Blockchain-Scan.
- **Verdrahtung (additiv, keine Reader/Broadcast-Änderung):**
  - Backfill bei Startup (`server.ts`, nach DB-Init) und am Ende von `runPayoutSync`
    (Abschnitt E, Reconcile).
  - Direkter kanonischer Write im Final-Payout-Resolver (`payoutSync` Abschnitt C):
    nutzt das ohnehin geladene `get_content` wieder und schreibt die *gesettleten*
    `net_votes`/`active_votes` + `last_payout` — **kein** zusätzlicher Chain-Call.

## Bewusst eingehaltene Grenzen

- Rein additive Migration; `vb_global_vote_outcomes`, `vb_copilot_shadow_runs`,
  `vb_whale_vote_details` und alle Spalten unverändert.
- Bestehende Leser unverändert. Legacy-Spalten werden weiter gepflegt.
- Vote-Broadcast-Pipeline (`recordVoteAtBroadcast`) nicht angefasst.
- Backfill ausschließlich aus vorhandenen Daten, kein Re-Scan.

## Für Phase 2/3 dokumentiert (nicht umgesetzt)

Bei der Umsetzung sichtbar gewordene Trennungen — vollständig in ADR-006:

1. `net_votes`/`active_votes` in gvo sind Vote-Zeitpunkt-Snapshots, keine gesettleten
   Post-Werte → Backfill daher dort meist NULL; Resolver-Write füllt die Settlement-Werte.
2. `paid_at` im Backfill ist Proxy (`MAX(realized_at)`); Resolver nutzt `last_payout`.
   Phase 2/3: durchgängig auf `last_payout`.
3. Drei Resolver laden dasselbe `get_content` (`payoutSync`, `shadowOutcomeResolver`,
   `whaleEnrichment`) → Phase 2 über `vb_post_outcomes` deduplizieren.
4. `community` dreifach gespeichert → Phase 3 konsolidieren.
5. Phase 2: `vb_model_predictions (model, voter, author, permlink)` — neue Modelle als
   INSERT statt ALTER TABLE; ersetzt `estimated_sp_*` und `v4_*`.
6. Phase 3: `vb_vote_outcomes` auf reines Vote-Outcome bereinigen.

# DevLog 2026-06-22 (Nachtrag) — Disk-Usage im System-Metrics-Widget

Admin will erkennen, **ob/wann der Server-Speicher vollläuft**. RAM/CPU/Uptime waren
schon im System-Tab; eine Festplatten-Metrik fehlte komplett.

## Was umgesetzt wurde

- **API** (`admin/routes.ts`, `/api/admin/system-metrics`): neues `system.disk`-Objekt
  via `fs/promises.statfs` auf das Verzeichnis der Live-DB (`getDbPath()` → das Volume,
  das tatsächlich vollläuft). Felder `totalGb, usedGb, freeGb, usedPct`. `used` ohne
  root-reservierten Anteil (= `df`-Sicht). Bei nicht unterstütztem `statfs` → `null`.
- **DB** (`db/index.ts`): `getDbPath()` exportiert (absoluter DB-Pfad).
- **Frontend** (`api.ts` Typ + `AdminDashboard.tsx`): neue **Disk**-Zelle unter RAM.
  Ampel: grün ≤75 %, gelb >75 %, rot >90 %. Bei `null` → "n/a".

## Verifikation

`tsc --noEmit` clean (api + web) · `statfs` im API-Container auf `/app/data`:
116,1 GB total / 36,3 belegt / 79,8 frei / 31,3 % — identisch zum Host-Volume.
Gebaut + deployed (web + api-dist via `docker cp`, API neugestartet).

## Offen / bewusst ausgeklammert

- Modell-/Shadow-Metriken gehören **nicht** ins System-Widget → Lab-Tab (später).
- Nächster Kandidat: rein infrastrukturelle DB-Spalte (SQLite-Größe, Tabellen, Zeilen,
  Growth/Tag); Fragmentierung erst später.

### Fix (gleiche Session)

Erster Deploy zeigte im Frontend "n/a": `disk` wurde berechnet, aber **nicht ins
zurückgegebene `system`-Objekt** aufgenommen. Nicht von `tsc` erkannt, da API-Response
und Web-`SystemMetrics`-Typ in getrennten Packages liegen (kein gemeinsamer Rückgabetyp).
`disk,` im Return ergänzt → am laufenden Endpoint verifiziert
(`{totalGb:116.1, usedGb:36.3, freeGb:79.8, usedPct:31.3}`).
