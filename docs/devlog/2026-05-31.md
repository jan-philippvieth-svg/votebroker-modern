### Bugfix: Vote-DNA API nicht erreichbar (Failed to fetch)

**Problem**

`api.ts` hatte `const API_BASE = import.meta.env.VITE_API_BASE ?? "http://localhost:3000"`. Die `.env`-Datei mit `VITE_API_BASE=/api` lag im Monorepo-Root `/opt/votebroker-modern/.env`, aber Vite liest `.env` nur aus dem Workspace-Verzeichnis `apps/web/`. Damit fiel der Build auf `http://localhost:3000` zurück – vom Browser aus nicht erreichbar. Der Fehler wurde zusätzlich durch `.catch(() => {})` still verschluckt, sodass der Tab einfach leer blieb.

**Umsetzung**

- `apps/web/src/api.ts`: Default von `"http://localhost:3000"` auf `""` geändert
  ```typescript
  const API_BASE = import.meta.env.VITE_API_BASE ?? "";
  ```
- Leerer String → alle Fetches als relative URL `/api/...` → nginx-Proxy greift korrekt
- `App.tsx`: `.catch(() => {})` auf `getCurationDna()` ersetzt durch `setCurationError(...)`
- Neue `curationError`-State-Variable; `CurationDnaPanel` zeigt jetzt sichtbaren Fehlertext statt leerem Tab

**Architekturentscheidung**

Relative URLs sind robuster als absolute URLs mit Environment-Variablen in Monorepo-Setups. nginx ist bereits korrekt konfiguriert (`location /api/ → proxy_pass http://api:3000/api/`). Keine `.env`-Dateien im Workspace-Verzeichnis nötig.

**Auswirkungen**

- Frontend: `api.ts` — API_BASE-Fallback
- Backend: keine
- Deployment: kein Rebuild der API nötig

**Offene Punkte**

- `.env.example` könnte den Hinweis "VITE_API_BASE muss in apps/web/ liegen" erhalten

### AuthorStats: Erweiterung mit Gewicht, Recency, Auswahlgründen

**Problem**

`topN(authorCounts, 15)` sortierte nur nach Häufigkeit. Autoren wie @steemchiller mit wenigen aber hochgewichteten Votes tauchten nicht auf. Kein Kontext warum ein Autor ausgewählt wurde. Keine Recency-Gewichtung.

**Umsetzung**

`packages/domain/src/curationDna.ts`:

```typescript
// Neues AuthorStats-Interface
export interface AuthorStats {
  username: string;
  voteCount: number;
  sharePct: number;
  avgWeightPct: number;      // neu: Durchschnittsgewicht dieser Autor-Votes
  compositeScore: number;    // neu: voteCount × avgWeightPct × recencyFactor
  lastVoteDaysAgo: number;  // neu: Alter des letzten Votes
  selectionReasons: string[]; // neu: erklärende Tags
}
```

Neue Berechnungslogik:
```typescript
// Per-Author Tracking
const authorWeightSum = new Map<string, number>();
const authorLastMs = new Map<string, number>();

// Composite Score mit Recency
const recency = Math.max(0.1, Math.exp(-lastVoteDaysAgo / 30)); // Halbwertszeit 30 Tage
const compositeScore = voteCount × avgWeightPct × recency
```

Neue Funktion `computeSelectionReasons()`:
- "Sehr hohe Aktivität" / "Regelmäßige Aktivität"
- "Konstant hohe Vote-Gewichte" / "Überdurchschnittliches Gewicht"
- "Sehr hoher Anteil aller Votes" / "Hoher Anteil aller Votes"
- "Kürzlich gevoted" / "Aktuelle Voting-Beziehung"
- "Langfristige Voting-Beziehung" (≥ 1×/Woche über Periode)
- "Strategisch wichtiger Autor" (avgWeight ≥ 70% AND voteCount ≥ 3)
- "Top-Autor in deiner Community"

Erhöhung von top-15 auf **top-50 Autoren**.
Sortierung nach `compositeScore` statt Häufigkeit.

`apps/web/src/api.ts`: AuthorStats-Interface synchronisiert.

**Architekturentscheidung**

Recency mit exponentieller Halbwertszeit 30 Tage: Votes die 30 Tage alt sind zählen noch 50%, nach 90 Tagen noch ~5%. Bewusste Entscheidung gegen hartcodierte Schwellwerte für "alt/neu". Das spiegelt reale Relevanz besser als binäre Cutoffs.

**Auswirkungen**

- Backend-Domain: `curationDna.ts` — AuthorStats, analyseCurationHistory
- API: Neues JSON-Shape für `/api/curation/dna` (2 neue Felder pro Autor)
- Frontend: `api.ts` (Typ), `App.tsx` (Nutzung in StrategyRuleRow)
- Benutzeroberfläche: Expandable Row zeigt Gründe als blaue Tags

**Offene Punkte**

- Consistency-Score: Wie gleichmäßig sind Votes über die Periode verteilt?
- Tag-basiertes Filtering (Votes auf bestimmte Tags höher gewichten)
- Community-spezifische Gewichtung

### Offene Votes — Open Vote Opportunities

**Problem**

VoteBroker generiert Strategie-Regeln, hilft dem Nutzer aber nicht, diese unmittelbar anzuwenden. Es gab keine Antwort auf: "Welche meiner Strategie-Autoren haben gerade neue Posts die ich noch nicht gevoted habe?"

**Umsetzung**

Backend — `apps/api/src/chain/recentPosts.ts` (neu):
- `fetchRecentPostsWithVotes(author, voter, limit)` — ruft `get_discussions_by_blog` vom Steem-Node ab
- Filtert Reblogs (nur eigene Posts des Autors)
- Prüft `active_votes` auf Voter-Username → `alreadyVoted`
- Berechnet Eligibility: nicht bereits gevoted, min. 5 Min. alt, max. 6.5 Tage alt (vor Payout-Lock)
- Gibt `PostOpportunity[]` zurück

Backend — `apps/api/src/curation/routes.ts` (erweitert):
- `POST /api/curation/opportunities` — nimmt `{ authors: string[], voterUsername: string }`
- Verarbeitet Autoren in 5er-Batches (`Promise.allSettled`)
- Gibt `{ opportunities: PostOpportunity[] }` zurück

Frontend — `apps/web/src/api.ts`:
- `PostOpportunity` Interface
- `getVoteOpportunities(authors, voter)` Funktion

Frontend — `apps/web/src/views/App.tsx`:
- State: `opportunities`, `opportunitiesLoading`, `opportunitiesError`
- `loadOpportunities()` — sammelt aktive Strategie-Autoren, ruft API ab
- `executeStrategyVotes()` — führt Vote-Sequenz mit 1.5s Delay aus (Steem-Rate-Limit)
- Neue Komponente `OpenVoteOpportunities` mit:
  - Per-Autor-Gruppen (Posts today / Gevoted / Offen)
  - Eligible Posts mit Titel, Alter, Gewicht
  - "Vote X offen" per Autor
  - "Vote alle N offenen Posts" global
  - Vote-Vorschau mit VP-Impact (aktuelle VP → VP nach Voten)
  - Bestätigung vor Ausführung
  - Ergebnis-Anzeige (✓ N Votes gesendet / ✗ N fehlgeschlagen)
- Neue Interface: `VoteTarget { author, permlink, title, ageMinutes, weightBps, category }`
- Neue Hilfsfunktion: `formatAge(minutes)` → "vor X Min./Std./Tag(en)"

**Architekturentscheidung**

Vote-Ausführung bleibt im bestehenden `POST /api/votes/execute` Endpoint (Session-Token required). Die Opportunities-Prüfung ist ein reiner Read-Endpoint ohne Auth. Der 1.5s Delay zwischen Votes ist konservativ (Steem erlaubt theoretisch mehr), aber schützt vor Rate-Limit-Fehlern.

**Auswirkungen**

- Backend: Neuer Chain-Modul `recentPosts.ts`, erweiterter Curation-Router
- API: Neuer Endpoint `POST /api/curation/opportunities`
- Frontend: Neue Section "Offene Votes" im DNA-Tab unter Strategy Editor
- Benutzeroberfläche: Vollständiger Workflow Vote-DNA → Strategie → Aktions-View

**Offene Punkte**

- Max-Alter-Konfiguration (aktuell fix 6.5 Tage)
- Auto-Refresh (z.B. alle 30 Min. im Hintergrund)
- Push-Notification wenn Lieblingsautor postet
- Curation-Window-Optimierung (beste 5-30 Min. nach Post)

### DevLog Content-System eingeführt

**Problem**

Technische Änderungen wurden intern dokumentiert, aber es gab keinen strukturierten Weg, daraus öffentliche Posts zu erzeugen. Posts wurden entweder nicht geschrieben oder mussten nachträglich erfunden werden.

**Umsetzung**

- `scripts/devlog-today.js` — Node.js ESM-Script, liest heutige DevLog-Dateien, generiert drei Draft-Dateien
- `docs/content/README.md` — Content-Richtlinien, Safety-Rules, Workflow-Dokumentation
- `package.json` — `npm run devlog:today` als neues Script

**Drei generierte Draft-Typen:**

1. `YYYY-MM-DD-product-post.md` — Für Steem-Nutzer: Was ist neu, was können Nutzer jetzt tun?
2. `YYYY-MM-DD-tech-post.md` — Für Entwickler: Architektur, Entscheidungen, Stack-Details
3. `YYYY-MM-DD-devlog-post.md` — Für Builder-Community: Was passierte, was brach, was lernten wir?

**Content Safety:**

15 Regex-Regeln blockieren automatisch: WIF-Keys, Hex-Tokens, Bearer-Tokens, Umgebungsvariablen mit Secrets, interne IPs.

**Architekturentscheidung**

Product-Post-Generator verwendet bewusst ein Skelett mit `<!-- ✏ EDIT -->` Markern statt automatischer Extraktion. Grund: DevLogs sind technisch geschrieben und können nicht automatisch in Nutzersprache übersetzt werden — das erfordert menschliches Urteil.

Tech-Post und DevLog-Post extrahieren direkt aus den Sections, da sie technisches Publikum haben.

**Auswirkungen**

- Workflow: Entwicklung → DevLog → `npm run devlog:today` → Review → Publish
- Niemals automatische Veröffentlichung
- `docs/content/` enthält nur Entwürfe

**Offene Punkte**

- Zukünftig: Versionierung der generierten Drafts
- Optional: Integration mit Steem-Blog-API für direktes Posting (nach manuellem Review)
